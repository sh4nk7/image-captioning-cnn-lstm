2. ORIGINAL ASSIGNMENT
=====================================================================

Assignment: Deep Learning and Generative Models - Project assignment #27

Objective:
- Given an input image, generate an image caption that describes its content.

Dataset:
- Flickr8k
- Source indicated by assignment:
  https://github.com/goodwillyoga/Flickr8k_dataset

Required network model:
- CNN-LSTM

Detailed assignment requirements:
- Build a CNN-LSTM architecture for image captioning
- Treat the task as image-to-sequence generation
- Extract visual features from an image using a CNN
- Pair the extracted image features with tokenized text descriptions during training
- Use an output decoder/MLP from LSTM hidden states to predict words
- Clean the text descriptions
- Convert words to lowercase
- Remove numbers
- Remove punctuation
- Remove single-letter words
- Save the processed dictionary/vocabulary in a file
- Use an encoding suitable for LSTM input

Interpretation of the assignment:
- The CNN is responsible for visual understanding
- The LSTM is responsible for language generation
- The final system must work end-to-end:
  image -> visual features -> word sequence -> caption

=====================================================================
3. HIGH-LEVEL PROJECT DESCRIPTION
=====================================================================

This repository implements an image captioning pipeline based on:
- InceptionV3 as image encoder
- LSTM as text decoder
- Flickr8k dataset for supervised learning
- BLEU metrics for evaluation
- optional Streamlit interface for interactive use

The intended complete flow is:

1. Prepare the dataset and captions
2. Clean the text and build a vocabulary
3. Build train/validation/test files
4. Train the CNN-LSTM model
5. Save checkpoints and TensorBoard logs
6. Evaluate the trained model with BLEU metrics
7. Generate qualitative visual examples
8. Optionally test the model in a Streamlit app

5.1 src/vocab_builder.py

Purpose:
- Clean Flickr8k captions
- Parse the original caption file format correctly
- Build the vocabulary
- Save the vocabulary to JSON
- Export processed train/validation CSV files
- Export test reference captions JSON

Inputs:
- Flickr8k caption file:
  data/Flickr8k_text/Flickr8k.token.txt
- Flickr8k split files:
  Flickr_8k.trainImages.txt
  Flickr_8k.devImages.txt
  Flickr_8k.testImages.txt

Outputs:
- data/vocab.json
- data/Flickr8k_text/train.csv
- data/Flickr8k_text/val.csv
- data/Flickr8k_text/test_captions.json

Why this file matters:
- It makes the preprocessing reproducible
- It guarantees the vocabulary is built from the real Flickr8k format
- It produces the exact files expected by training and evaluation


5.2 src/prepare_data.py

Purpose:
- Convenience wrapper around vocabulary creation and split export

Why this file matters:
- Reduces human error
- Allows a single reproducible command for all data preparation

Typical command:
- python src/prepare_data.py


5.3 src/dataset.py

Purpose:
- Load images and captions for training/validation
- Convert cleaned captions into token IDs
- Apply image transforms
- Pad variable-length sequences through a custom collate function

Inputs:
- processed CSV files
- vocabulary JSON
- image directory

Outputs:
- batches of images and token sequences for PyTorch training


5.4 src/model.py

Purpose:
- Define the neural architecture

Components:
- EncoderCNN:
  uses pretrained InceptionV3 and projects visual features to embedding space
- DecoderLSTM:
  receives image-conditioned input and predicts word tokens step by step
- CNNtoLSTM:
  wraps encoder + decoder in a single model

Important implementation note:
- During revision, the encoder was corrected so that CNN fine-tuning can
  actually happen when enabled, instead of being blocked by no_grad.


5.5 src/train.py

Purpose:
- Train the CNN-LSTM captioning model

Features included:
- reproducibility seed
- DataLoader setup
- scheduled sampling support
- cross-entropy loss with label smoothing
- gradient clipping
- validation loss computation
- checkpoint saving
- TensorBoard logging
- resume from checkpoint support

Important implementation note:
- During revision, the training loss alignment was corrected so that the
  model learns next-token prediction properly instead of incorrectly scoring
  the initial special timestep associated with image features.


5.6 src/eval.py

Purpose:
- Load a trained checkpoint
- Generate captions for test images
- Compare predictions against 5 reference captions per image
- Compute BLEU-1, BLEU-2, BLEU-3, BLEU-4
- Save predictions and metric results

Why this file matters:
- It provides the quantitative evidence required for an academic project


5.7 src/visualize.py

Purpose:
- Generate qualitative examples by saving images with predicted and reference captions

Why this file matters:
- It supports discussion and qualitative error analysis


5.8 src/ui_streamlit.py

Purpose:
- Provide a small interactive interface for testing the model on uploaded images

Why this file matters:

- Not required for core assignment correctness, but valuable as demo material


5.9 src/utils.py

Purpose:

Current role:

8. MAJOR PROBLEMS IDENTIFIED DURING ANALYSIS
=====================================================================

Before revision, the repository had several critical issues.

Problem 1:
- README and code referenced files that did not exist

Examples:
- requirements.txt missing
- train.csv missing
- val.csv missing
- test_captions.json missing
- config.yaml referenced even though config.yml existed

Impact:
- The repository was not reproducible end-to-end


Problem 2:
- Vocabulary builder parsed captions incorrectly

Cause:
- The original code split lines on commas, but Flickr8k captions are tab-separated

Impact:
- Vocabulary generation was not truly reproducible from source data
- Large parts of the dataset could be skipped or parsed incorrectly


Problem 3:
- CNN fine-tuning was effectively disabled even when configured as enabled

Cause:
- Encoder forward path used no_grad and forced evaluation behavior

Impact:
- Claims about fine-tuning were not aligned with actual computation


Problem 4:
- Training loss alignment was wrong

Cause:
- Decoder outputs and target tokens were not shifted correctly

Impact:
- The model could be optimized against misaligned targets


Problem 5:
- Evaluation/UI used default paths or assumptions inconsistent with actual files

Impact:
- Evaluation and demo were harder to reproduce reliably


Problem 6:
- Scheduler compatibility issue with local runtime

Observed runtime error:
- ReduceLROnPlateau received an unexpected verbose argument

Impact:
- Training crashed at runtime before starting real optimization
